# Руководство: M3 Expressive на базе React Aria для проекта Flow

Это руководство предназначено для ИИ-агента, который создаёт библиотеку React-компонентов для Flow.  
Цель — строить интерфейс, совместимый с канонами M3 Expressive и адаптированный под философию Flow, используя доступную логику React Aria, утверждённые design tokens и правила calm UI.

---

## 1. Фундаментальные принципы (каноны)

Для стабильного и предсказуемого поведения нужно строго разделять логику и представление.

- Логика: используй `react-aria-components` как основу. Для сложных случаев используй низкоуровневые хуки `react-aria`.
- Стилизация: не используй неутверждённые ad hoc-стили. Все значения берутся только из Design Tokens Flow, CSS-переменных и разрешённых utility-классов.
- Состояния: стилизуй компоненты через `data-*` attributes, которые предоставляет React Aria.
- Доступность: интерактивные зоны должны быть удобны для касания, клавиатуры и скринридера; visible focus rings и корректные labels обязательны.

---

## 2. Адаптивная эргономика (критическое правило Flow)

Любая компоновка экрана должна учитывать глобальный контекст `isLeftHanded`, если он включён.

- FAB и акценты: главная плавающая кнопка динамически позиционируется в нижнем углу в зависимости от `isLeftHanded`. Жёсткая фиксация по центру или справа запрещена.
- Жесты (Swipe): направление критических свайпов инвертируется по оси X, если это улучшает reach и снижает трение.
- Контекстные меню: не привязывай открытие меню строго к правому краю карточки; меню должно открываться в безопасной зоне и не перекрываться рукой пользователя.
- Если адаптация для левши конфликтует с читаемостью, приоритет у доступности и ясности, а не у зеркалирования.

---

## 3. Система цвета и ролей

M3 задаёт базовые роли, а Flow добавляет собственные токены для задач.

### Служебный UI (M3 tokens)
- Primary: главное действие, например Filled button.
- Primary Container: фон для primary-элементов.
- Secondary: второстепенные действия, фильтры, чипы.
- Surface: фон карточек, модалок и панелей.

### Карточки задач (Flow custom tokens)
- Quick: `flow-weight-quick`, контент `flow-weight-quick-on`.
- Focused: `flow-weight-focused`, контент `flow-weight-focused-on`.
- Deep: `flow-weight-deep`, контент `flow-weight-deep-on`.

Запрещено использовать системные цвета для кодирования веса задач, если для этого уже есть утверждённые Flow tokens.

---

## 4. Спецификации компонентов

### 4.1 Формы задач и анатомия
Форма карточки во Flow кодирует только вес задачи, а не приоритет.

- Quick Tasks: круглая форма, минимальная визуальная нагрузка, текст не обязателен.
- Focused Tasks: мягкая квадратная форма, допускается короткий текст.
- Deep Tasks: более сложная органическая форма, допускается и текст, и иконка.
- Острые углы не являются дефолтом. Допускаются только если они явно улучшают смысл, иерархию или читаемость.

### 4.2 Кнопки
Используй `react-aria-components/Button` или `useButton`.

- Обрабатывай действия через `onPress`, а не через нативный `onClick`, если нужен единый доступный паттерн.
- Voice FAB является главным персистентным элементом ввода, если voice-режим активен. Он либо заменяет стандартный Add FAB, либо переносится внутрь Add Task flow, чтобы избежать конфликта двух плавающих кнопок.

### 4.3 Формы и поля ввода
Используй `TextField` из React Aria.

- Всегда резервируй место под вспомогательный текст, ошибки валидации и подсказки.
- Избегай layout shift.
- Используй `useFocusRing`, чтобы фокус был виден только при клавиатурной навигации.

---

## 5. Движение и физика

Пространственные переходы должны быть предсказуемыми и не создавать лишней когнитивной нагрузки.

- Для UI transitions используй утверждённые tokens и motion presets проекта.
- Не вводи сторонние spring-библиотеки для экранных переходов и панелей, если они не были отдельно утверждены.
- Matter.js применяется только внутри циркадных блоков и только там, где физика действительно усиливает понимание сцены.
- Если включён Recovery Mode или `prefers-reduced-motion: reduce`, физика отключается, а сложные анимации заменяются на короткий fade-in.

## 6. Проверочные вопросы для ИИ-агента

Вопрос: Какую форму и цвет должна иметь задача с высоким приоритетом во Flow?  
Ожидаемый ответ: Приоритет не кодируется формой или острыми углами. Форма кодирует вес задачи, а цвета берутся из `flow-weight-*` токенов с контрастным `*-on` для содержимого.

Вопрос: Куда позиционируется кнопка FAB на главном экране?  
Ожидаемый ответ: FAB динамически позиционируется в нижний угол в зависимости от `isLeftHanded`. Жёсткая фиксация по центру или справа запрещена.

Вопрос: Как ты анимируешь появление модального окна настроек?  
Ожидаемый ответ: Используя утверждённые motion tokens проекта. При активном Recovery Mode анимация схлопывается до короткого fade-in.

Вопрос: Используешь ли ты `useFocusRing` для стилизации фокуса?  
Ожидаемый ответ: Да, чтобы фокус был виден только при использовании клавиатуры.

Вопрос: Как ты обрабатываешь события нажатия на кнопки?  
Ожидаемый ответ: Через `onPress` и доступные паттерны React Aria, а не через случайные нативные обработчики без контекста доступности.
